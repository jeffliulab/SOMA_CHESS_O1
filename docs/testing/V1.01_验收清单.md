# V1.01 Week 1 + Week 2 统一测试与验收

> 适用范围：`V1.01 = Week 1 + Week 2`
>
> 当前版本只验收：
> - `W1.4A` 基线锁参与稳定成像
> - `W1.6` 相机内参与 eye-to-hand 标定
> - `W1.8` 工作区可达性验证
> - `W1.4B` 图像质量驱动相机自适应 v1
> - `Week 2` geometry-first `/find_object` 与 `camera_feedback`
>
> 当前版本**不**验收：
> - MoveIt2 真机原语闭环
> - ACT 数据采集与训练
> - ANIMA 端到端闭环

---

## 1. 验收目标

这轮统一测试的目标不是“把所有未来功能都跑一遍”，而是收掉 `Week 1 + Week 2` 的基础闭环：

1. 相机在当前工作站上可以稳定输出，不会每次重启都漂。
2. calibration 三件套能从真实图像产出正式 YAML：
   - `camera_intrinsics.yaml`
   - `eye_to_hand.yaml`
   - `workspace.yaml`
3. 工作区 reachability 结果可以被正式记录并回写。
4. `/find_object` 能对棋盘、容器、通用桌面前景目标给出可信的像素和世界坐标。
5. `camera_feedback` 与 Windows adaptive sender 的通路保持正常。

## 2. 统一通过标准

本轮通过标准分成五类。

### A. W1.4A 相机基线

- 连续 `3` 次冷启动后，画面亮度、白平衡、焦点没有明显跳变。
- 连续观察 `2` 分钟，手或机械臂短时进出画面时，不出现明显的自动曝光抽动。
- WSL 侧 `/camera/image_raw` 持续可用，`/camera/camera_info` 正常发布。

### B. W1.6 标定

- `calibrate_camera_charuco.py` 能写出正式 `camera_intrinsics.yaml`
- `reprojection_error_px < 5`
- `solve_workspace_calibration.py` 能写出正式 `eye_to_hand.yaml + workspace.yaml`
- `workspace.yaml` 内至少有：
  - `board_roi_norm`
  - `pixel_to_world_homography`
  - `tabletop_polygon_xy_m`
  - `reachable_test_points_xy_m`

### C. W1.8 reachability

- 四角、中心、容器区都完成 `5` 次真实测试。
- 测试结果已回写进 `workspace.yaml`
- 若存在失败点，`no_go_regions_xy_m` 已同步生成。

### D. Week 2 geometry-first perception

- `/find_object` 能响应：
  - `board / chessboard / workspace`
  - `bin / box / container`
  - `piece / object`
- 返回的 `pixel_x / pixel_y` 落在合理区域。
- `world_x / world_y` 与真实工作区位置一致到“可用”水平。

### E. W1.4B adaptive

- `adaptive` 模式在 3 类场景下不持续振荡：
  - 台灯单独开
  - 混合室内光
  - 短时遮挡
- 关闭 adaptive 后，sender 行为能回到 `locked baseline`

## 4. 推荐测试顺序

统一测试时，严格按下面顺序进行，不要跳着测。

1. `teleop / 驱动` 基线确认
2. `W1.4A` 锁参与稳定成像
3. `W1.6` 相机内参
4. `W1.6` workspace + eye-to-hand
5. `W1.8` reachability
6. `Week 2` `/find_object`
7. `W1.4B` adaptive sender

原因很简单：

- 没有稳定相机，就不该做标定。
- 没有正式 calibration YAML，就不该认真测 `/find_object`。
- 没有 locked baseline，就不该认真评估 adaptive。

## 5. Phase 0：teleop / 驱动基线确认

目标：确认不是硬件连接或 ROS 基础链路出了问题。

### 步骤

```bash
cd ~/SOMA/soma-arm
scripts/start_teleop_wsl_gamepad.sh
```

启动后：

1. 等待 `/joint_states`
2. 按一次 `Start`
3. 确认机械臂能响应手柄输入

### 通过标准

- 日志出现 `Teleop target initialized from current /joint_states.`
- 日志出现 `Start pressed — re-enabled, going home`
- 摇杆和夹爪控制正常

### 如果失败

- 没有 `/dev/ttyUSB*`：检查 RoArm 是否 attach 到 WSL
- 没有 `/dev/input/event*`：检查手柄 attach 和 input 支持
- 有设备但无 `/joint_states`：回查 `arm_driver`

## 13. 现场故障定位顺序

如果统一联调失败，不要乱改，按这个顺序排：

1. **硬件接入**
   - `usbipd list`
   - `/dev/ttyUSB*`
   - `/dev/input/event*`
2. **teleop / 驱动**
   - `/joint_states`
   - 手柄 `Start`
3. **相机基础链路**
   - Windows sender 是否正常开
   - WSL receiver 是否正常发 `/camera/image_raw`
4. **locked baseline**
   - 先确认画面稳定，再谈标定
5. **calibration YAML**
   - 是否还是 `pending_calibration`
6. **perception**
   - `/find_object` 是否真的读到了正式 `workspace.yaml`
7. **adaptive**
   - 永远最后看，不要在基础链路不稳时先怪 adaptive
